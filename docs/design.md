# JobLens · V1 终态设计

> 这份文档定义 V1 完成时项目"长什么样"，是产品蓝图，不是实现细节。
> 任何开发期的决策，先用这里的"设计原则"和"V1 范围"对照，再动手。

---

## 一、一句话定位

> 把"一份简历 + 一个 JD"在 30 秒内变成一份**带评分、带改写、带模拟面试**的求职行动方案。

差异点：不是又一个"AI 帮你润色简历"的工具——是**多 Agent 并行可视化**的工作台，面试官打开 demo 链接就能看到四个 AI 角色在屏幕上同时工作。

---

## 二、用户旅程（核心 demo 路径）

```
[落地页]
   │   ① 不需要登录，直接体验
   ▼
[输入页]
   │   左：粘贴 JD 文本框（也可贴 URL,V2 自动抓取）
   │   右：上传简历（PDF/MD/纯文本）或点"用示例简历"
   │   底部一个大按钮：开始分析 →
   ▼
[分析中页面 — 这是 demo 的高光时刻]
   │   屏幕分成 4 个面板，4 个 Agent 并行流式工作
   │   每个面板顶部显示 Agent 头像 + 名字 + 状态（思考中/已完成）
   │   面板里是流式输出的文字
   │   平均 8-15 秒全部完成
   ▼
[结果页 — 单屏可滚动报告]
   │   顶部 hero：综合匹配分（大字数字 + 等级 S/A/B/C）
   │   ↓
   │   雷达图：5 维度评分
   │   ↓
   │   关键词覆盖热力图：JD 关键词 × 简历命中
   │   ↓
   │   逐句改写 diff：原文 / 改写 / 改写理由
   │   ↓
   │   模拟面试问题卡片：3-5 张
   │   ↓
   │   底部：导出 PDF / 分享链接 / 再来一次
```

**整个 demo 现场演示时长：约 60-90 秒。**

## 四、Agent 设计

| Agent | 输入 | 输出（JSON schema） | 模型 |
|---|---|---|---|
| `JDParserAgent` | JD 原文 | `{ hard_skills[], soft_skills[], hidden_requirements[], seniority, keywords[] }` | Haiku 4.5 |
| `ResumeAnalystAgent` | 简历结构化 JSON | `{ highlights[], weaknesses[], experience_years, domain_tags[] }` | Sonnet 4.6 |
| `MatchScorerAgent` | 上面两个的输出 | `{ overall_score, dim_scores{tech, experience, project, communication, uniqueness}, summary }` | Sonnet 4.6 |
| `RewriterAgent` | 简历 bullets + JD 要求 | `[{ original, rewritten, reason, impact: 1-3 }]` | Sonnet 4.6 |
| `InterviewerAgent` | 简历 + JD + 弱项分析 | `[{ question, probe_point, suggested_angle }]` | Sonnet 4.6 |

**编排关系：**
```
                 ┌→ JDParserAgent ─┐
[用户输入] ──→ │                  ├→ MatchScorerAgent ─┐
                 └→ ResumeAnalyst ──┤                    ├→ [结果页]
                                    ├→ RewriterAgent ───┤
                                    └→ InterviewerAgent ┘
```

前两个并行启动，后三个依赖前两个完成后并行启动。

## 五、设计原则（拍板锚点）

1. **演示优先于易用** —— 任何决策先问"对 30 秒 demo 有帮助吗？"
2. **可视化不是装饰，是产品本身** —— 雷达图、diff、热力图、四面板流式，是和其他 AI 简历工具的核心差异
3. **不做账号系统也能用** —— 落地体验零摩擦；账号只为"保留历史"加分项
4. **可讲性最大化** —— 每个模块要能用一句话讲清"为什么这么设计"
5. **不和 ChatGPT/Kimi 对比通用对话能力** —— 我们卖的是**结构化输出 + 可视化**，不是模型本身

## 六、已拍板的设计选择（V1）

| # | 选项 | 决定 |
|---|---|---|
| 1 | 视觉风格 | **极简白底**（Linear / Vercel 风格），主色调白 + 浅灰 + 一个克制的强调色（待定，建议偏冷蓝或墨绿） |
| 2 | 目标语言 | **中文优先**：中文 JD + 中文简历是默认场景；UI 中文；英文支持作为 V2 |
| 3 | 示例数据 | **必须有**：内置一份"假简历 + 假 JD"，落地页一键"用示例体验"，零门槛 demo |
| 4 | 商业化 | **V1 完全免费**：不做付费墙，不做导出限制，全功能开放 |

## 七、V1 范围 vs V2 留白

| 模块 | V1 必做 | V2 留白 |
|---|---|---|
| 输入 | JD 文本 + 简历 PDF/MD + 内置示例 | JD URL 自动抓取（LinkedIn/拉勾/Boss） |
| 解析 | 简历→结构化 JSON | OCR 扫描件、图片简历 |
| Agent | 5 个 Agent 并行流式 | Agent 之间互相辩论、多轮迭代 |
| 评分 | 5 维度雷达 | 行业基准对比、岗位类目专属 rubric |
| 改写 | 逐 bullet diff + 改写理由 + 影响力排序 | 整段重写 / 多版本 A/B |
| 面试 | 3-5 问题卡 + 考察点 + 回答角度提示 | 真·语音模拟面试 / 多轮追问 |
| 账号 | 匿名一次性 + 分享链接 | 历史记录 / 多简历库 / Google 登录 |
| 导出 | PDF（免费） | Notion / Word / 简历回写 |
| 多语言 | 中文（UI + JD + 简历） | 英文、日韩 |
| 隐私 | 默认 24h 后删除内容 + 首页明确告知 | 本地 Ollama 模式 |

## 八、示例数据（必备）

设计阶段就准备好这份"假简历 + 假 JD"，开发期所有测试、demo 视频、面试现场演示都用它。

- **假简历主角**：一个 3 年经验的后端工程师"张某"，简历有明显可改进点（缺指标、缺规模数字、技术栈写得太模糊）
- **假 JD**：一份"高级后端工程师 / LLM 应用方向"的 JD，要求里有 60% 和张某匹配、30% 弱匹配、10% 缺失——这样跑出来的雷达图、diff、面试问题都有戏剧效果
- 文件位置：`fixtures/demo-resume.md` + `fixtures/demo-jd.md`，仓库提交

## 九、面试 talking points 预埋

设计阶段就把这些"埋"进去，面试时能讲：

- **为什么 5 个 Agent 而不是一个大 prompt？** → 单 Agent 输出长上下文易降质；多 Agent 各司其职 + 并行 = 更快 + 更准 + 视觉冲击
- **Agent 之间怎么传递信息？** → 共享一个结构化的"分析上下文对象"，每个 Agent 读它需要的字段、写它生产的字段——不是把全部原文丢给每个 Agent
- **怎么防止 Agent 输出不稳定？** → 每个 Agent 输出严格 JSON schema + Zod 校验 + 失败重试一次
- **成本怎么算？** → 一次完整分析约 X 美分，prompt caching 命中后 Y 美分（开发阶段实测填入）
- **隐私怎么做？** → 简历内容仅在调用 Claude 时发送，本地 24h 后删；可加"本地 Ollama 模式"留 V2
- **为什么用 Claude 而不是 GPT？** → Sonnet 4.6 长上下文 + 结构化输出稳定 + prompt caching 成本优势

## 十、成功标准（V1 上线时要达到）

- [ ] 在无缓存的冷启动下，从点"开始分析"到结果页首屏可见，**≤ 15 秒**
- [ ] 落地页有可自动播放的 **30 秒 demo 视频**
- [ ] 任意访客（无账号）可以**用示例数据一键体验**
- [ ] 结果页可生成**可分享的公开链接**（用作面试现场打开）
- [ ] 部署在**自有域名**（HTTPS，可访问性 > 99%）
- [ ] README 中有：架构图（Mermaid）、技术决策、成本估算、踩坑记录
- [ ] 一篇技术博客（中文）讲清"我是怎么做这个的"
